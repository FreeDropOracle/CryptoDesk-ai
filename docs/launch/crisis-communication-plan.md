# Crisis Communication Plan

Use this document when a beta incident is serious enough to require structured communication.

## Principles

- acknowledge quickly
- avoid speculation
- give the next concrete update time
- keep security-sensitive details out of public channels until verified

## Scenario: Security Vulnerability

Target response time: under 4 hours

Actions:

1. Pause beta distribution if needed
2. Notify active testers through the highest-priority channel
3. Share immediate safety guidance
4. Publish the next update window
5. Prepare a follow-up summary after mitigation

## Scenario: Major Data Loss Or Corruption

Target response time: under 2 hours

Actions:

1. Acknowledge the issue publicly to the beta cohort
2. Explain the affected scope as clearly as possible
3. Share recovery steps if any exist
4. Pause risky workflows until the cause is understood
5. Follow up with a prevention plan

## Scenario: Long Outage Or Broken Build

Target response time: under 1 hour

Actions:

1. Confirm the outage or release problem
2. Share affected platforms and scope
3. Give an ETA only if confidence is reasonable
4. Post regular updates until resolved
5. Log the incident in the beta retrospective

## Communication Channels

- Email for high-priority cohort-wide messages
- Discord for real-time updates
- Release notes or changelog for documented fixes
- Public social channels only if the incident is externally visible

## Minimum Message Format

- what happened
- who is affected
- what users should do now
- when the next update will arrive
